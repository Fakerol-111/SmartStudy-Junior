# SmartStudy 架构设计

## 一、模块总览

| 模块 | 职责 |
|------|------|
| **Router AI** | 独立的路由模型，接收学生输入后判断意图，分发给对应的 Handler。负责意图识别、学科识别、话题切换判断 |
| **Harness** | 状态机引擎，根据 Router 的分发结果控制各 Handler 的"生成→审查→教学"流转 |
| **DeepSeek Client** | LLM 调用封装，支持快速/旗舰双模型切换 |
| **Tool Registry** | 工具注册中心，教学者与审查者各有独立的注册实例 |
| **Prompt Manager** | 学科 Prompt 的查找/缓存/AI 自生成 |
| **Reviewer** | 审查者独立对话管理，负责 Generate 和 Teach 阶段的输出审核 |
| **Memory Manager** | 记忆库管理：错题本、掌握程度、卡住记录、学生画像、教师 Soul |
| **Student Profile** | AI 每 5 次对话自动维护的学生能力画像，拼接在 System Prompt 中 |
| **Teacher Soul** | AI 持续维护的教师人格画像，记录教学风格、沟通偏好、情感适配策略，拼接在 System Prompt 中 |
| **Knowledge Blacklist** | 学生主动标记的"超纲"知识点黑名单，审查者据此拦截，用越多越准 |

Handler 说明:

| Handler | 职责 |
|---------|------|
| **Teach Handler** | 分步引导教学，使用学科 prompt + 学生画像 + 教师 Soul |
| **Verify Handler** | 快速批改：判断对错、指出错因、询问是否需要步骤讲解 |
| **Concept Handler** | 概念讲解：直接解释知识点，不进入 Teach 循环 |
| **Review Handler** | 复习：查询记忆库薄弱点，生成复习计划 |
| **Correction Handler** | 纠错：学生指出 AI 错误时，验证并修正 |
| **Feedback Handler** | 情感与反馈处理：学生不耐烦、觉得空洞、风格不合适时，更新 Teacher Soul |

---

## 二、模型分层

| 层级 | 说明 |
|------|------|
| **快速模型（默认）** | 所有阶段默认使用。大多数初中问题足以应对 |
| **旗舰模型（可选）** | 审查者判断快速模型输出质量不足时升级调用。**每阶段仅升级一次**，再不通过则降级 |

模型配置通过 `.env` 环境变量注入，不写死在代码中。最少只需配置一套默认模型即可工作。

## 三、回复一道问题的工作流

```
┌─ ① 学生输入问题
│
├─ ② Router AI — 意图识别与分发
│    接收学生输入，返回分发结果:
│    { intent, subjects, newTopic, targetHandler }
│
│    → 前端显示: "让我看看你问的是什么……"
│
│    根据意图分流:
│    ├─ teach    → 发给 Teach Handler → 完整教学流程（③~⑥）
│    ├─ verify   → 发给 Verify Handler → 快速批改流程
│    ├─ concept  → 发给 Concept Handler → 概念讲解流程
│    ├─ review   → 发给 Review Handler → 复习流程
│    ├─ correct  → 发给 Correction Handler → 纠错流程
│    ├─ feedback → 发给 Feedback Handler → 更新 Soul
│    └─ unknown  → 兜底处理：提示学生重新表述
│
│    newTopic 附加在任意 intent 上，不是独立的意图:
│    ├─ newTopic=true  → 归档当前对话，清空 history，创建新对话
│    └─ newTopic=false → 继续当前对话
│
├─ ③~⑥ — 完整教学流程

完整教学流程（用于 intent=teach）:
│
├─ ③ Generate — 生成教学回答
│    → 前端显示: "原来是物理题，让老师想想……"
│    快速模型生成 → 审查者判断质量 →
│    ├─ 合格 → 继续
│    └─ 不合格 → 切旗舰模型重生成
│         → 前端显示: "这个老师不太行，再为您换一个老师……"
│
├─ ④ Review — 前置审查（独立对话）
│    → 前端显示: "老师检查一下这样讲合不合适……"
│    审查者审查完整教学方案 →
│    ├─ 通过 → 进入 Teach
│    └─ 不通过 → 返回 Generate 重做
│
├─ ⑤ Teach — 分步引导教学（循环）
│    每步: 生成教学输出 → 审查 → 通过 → 展示给学生 → 保存到 DB →
│    → 前端显示: "来，我们一步一步看……"（第一步）
│    → 前端显示: "这一步你理解了吗？"（每步确认）
│    确认学生理解 →
│    ├─ 理解了 → 下一步
│    ├─ 不理解 → 换角度再讲一遍
│    │   └─ 卡住时 AI 结合学生画像判断：
│    │       ├─ 薄弱点 → "这个地方你上次也不太熟，我们换种方式讲"
│    │       └─ 偶发 → "没关系，我们换个角度看看"
│    └─ 完成 → 退出循环
│
├─ ⑥ 保存记忆库
│    → 前端显示: "老师记下来，下次复习的时候提醒你"
│    错题记录 + 卡住知识点 + 掌握程度更新
│    └─ conversation_count++ → 达到 5 时 AI 自动更新学生画像 + Soul
│
└─ 回到 IDLE

快速批改流程（用于 intent=verify）:
│
├─ ③ AI 快速判断答案对错（默认快速模型）
│    → 前端显示: "让老师看看你算得对不对……"
│    ├─ 正确 → "完全正确！👍"
│    │   └─ 询问"需要帮你看看步骤吗？" → 是则进入 Teach，否则结束
│    └─ 错误 → "这里有个小问题，你看……"
│        └─ 指出错误位置 → 引导纠正 → 进入简化 Teach
│
└─ 回到 IDLE

概念讲解流程（用于 intent=concept）:
│
├─ ③ AI 直接讲解概念（默认快速模型，需审查）
│    → 前端显示: "这个概念的来龙去脉是这样的……"
├─ ④ 确认学生理解 →
│    ├─ 理解了 → "懂了就好！"
│    └─ 没理解 → "那我们换个角度来讲……"
│
└─ 回到 IDLE

复习流程（用于 intent=review）:
│
├─ ③ 查询记忆库中薄弱知识点 → 生成复习计划
│    → 前端显示: "老师看看你之前哪些地方不太熟……"
│    → 前端显示: "找到几个需要巩固的点，我们一个个来看"
│    → 进入 Teach 循环
│
└─ 回到 IDLE

纠错流程（用于 intent=correct）:
│
├─ ③ 接收学生指出的错误内容
│    → 前端显示: "让老师看看是不是讲错了……"
├─ ④ 验证 AI 之前回答是否正确
│    ├─ 确实错了 → "你说得对，是老师讲错了，抱歉！"
│    │   → 给出修正 → 审查 → 通过后输出
│    └─ 没错 → "老师检查过了，这个步骤是对的，你再看这里……"
│
└─ 回到原 Handler

反馈流程（用于 intent=feedback）:
│
├─ ③ 检测学生情感信号
│    → 前端显示: "老师知道了，下次注意……"
├─ ④ 把旧 Soul + 当前对话历史丢给 AI，让 AI 自行分析并更新 Soul
├─ ⑤ 保存新 Soul 到 DB → 用调整后的风格继续
│
└─ 回到 IDLE
```

## 四、各阶段细节

### 4.1 Router AI

```
Router AI 是一个独立模型实例（默认使用快速模型），使用极简 Prompt，
只做一件事：判断学生输入属于哪个意图，然后分发给对应的 Handler。

┌─ Router AI 工作流程:
│
│  学生输入 → Router AI →
│  { intent, subjects, newTopic, targetHandler }
│
│  intent 类型:
│  ├─ teach    → "这道题怎么做" / "帮我讲一下"
│  ├─ verify   → "我算出来是 42，对吗？" / "是不是选 C"
│  ├─ concept  → "什么是勾股定理" / "解释一下"
│  ├─ review   → "帮我出几道题" / "我想复习"
│  ├─ correct  → "你这一步错了" / "不是这样算的" / "答案是 5 不是 3"
│  ├─ feedback → "你讲得太啰嗦了" / "能不能直接说重点" / "我还是听不懂"
│  └─ unknown  → 乱码 / 空白 / 完全无关的内容
│
│  newTopic 不是独立的意图，而是附加在任意 intent 上的属性。
│  例如: "先不说这个，帮我看看这道英语题"
│       → { intent: "teach", subjects: ["english"], newTopic: true }
│  而不是 { intent: "switch_topic", ... }
│
│  subjects: 学科列表（如 ["math", "physics"]）
│  newTopic: 是否开启新对话
│  targetHandler: 对应 Handler 的名称
│
│  发给 Handler 时携带:
│  - 学生原始输入
│  - 学科列表
│  - 学生画像 + 教师 Soul（从 System Prompt 读取）
│  - 当前对话 history（仅同话题时）

最终 System Prompt:
  [BASE] + [学科指令] + [学生画像] + [教师 Soul] + [标记协议] + [年级信息] + [补充指令]

Router AI 的 Prompt 示例（约 100 tokens）:
  "判断以下学生输入属于哪个意图。
   可选: teach, verify, concept, review, correct, feedback, unknown。
   只输出意图名，不要其他内容。"
```

### 4.2 各 Handler

```
Teach Handler:
  使用学科 Prompt + 学生画像 → Generate → 审查 → 分步教学循环
  （即原有的完整教学流程）

Verify Handler:
  简短 Prompt，直接判断答案对错 →
  ├─ 正确 → 确认 + "需要帮你看看步骤吗？"
  └─ 错误 → 指出错因 → 进入简化 Teach（仅针对错因）

Concept Handler:
  直接讲解概念 → 需审查 → 通过则输出，不通过则切旗舰重生成

Review Handler:
  查询 memory_manager 中薄弱知识点 → 生成复习计划 → 进入 Teach

Correction Handler:
  学生指出 AI 错误时的专用 Handler →
  ① 接收学生指出的错误内容
  ② 调用 AI 验证自己之前的回答是否正确
  ③ ├─ 确实错了 → 承认错误 → 给出修正后的答案 → 重新审查
     └─ 没错 → 解释为什么没错 → 继续原教学
  → 修正完成后回到原 Handler 继续

Feedback Handler:
  学生表达不满或情感反馈时的专用 Handler →
  ① 接收学生输入，提取情感信号（显式或隐式）
     ├─ 显式: "太啰嗦" / "太复杂" / "讲得不对路"
     └─ 隐式: 多次说"还是不懂" / 语气变差 / 重复同一疑问
  ② 把旧 Soul + 当前对话历史丢给 AI，让 AI 自行分析并更新 Soul
  ③ 保存新 Soul 到 DB
  ④ 回到原 Handler，用调整后的 Soul 风格继续教学

Unknown 兜底:
  Router AI 无法识别意图时，返回 "我不太明白你的意思，
  能换个说法吗？" 并回到 IDLE。不触发任何 Handler。
```

### 4.3 审查者

```
独立对话，与教学者互不干扰。
两个审查时机:
  - 前置审查: Generate 阶段结束后，审查完整教学方案
  - 实时审查: Teach 阶段每步输出前，审查这一步内容

审查维度: 超纲 / 直接给答案 / 知识错误 / 不适龄 / 模糊 / 跳步 / 离题

超纲检测:
  不依赖预设的知识点白名单，由审查者凭 AI 自身知识判断是否超出初中范围。
  同时查询学生个人黑名单作为辅助依据:
  ├─ 在黑名单中 → 直接判定为超纲
  └─ 不在黑名单中 → 审查者自行判断
  → 随着黑名单积累，超纲拦截越来越准确

触发黑名单记录:
  Teach 阶段中学生明确说"这不是我要学的" / "这个不考" / "超出范围了"
  → AI 记录该知识点到学生黑名单
  → 后续审查自动拦截该知识点
```

### 4.4 学生画像

```
维护者: AI 自己
更新频率: 每 5 次对话更新一次
存储位置: students 表 profile 字段（JSON）
画像内容: AI 自由写的文字描述 + 各学科能力评分（0~100）
使用方式: 每次对话开始时拼接到 System Prompt
作用: AI 根据画像自动调整教学策略（薄弱点直接给提示、已掌握快速通过）
```

### 4.5 Teacher Soul（教师人格）

```
Soul 是教师的"人格画像"，记录教师应该以什么风格和态度面对这个学生。
它与 Student Profile 并列，一起影响所有 Handler 的输出。

| 对比 | Student Profile | Teacher Soul |
|------|----------------|--------------|
| 描述谁 | 学生 | 老师自己 |
| 内容 | 能力、薄弱点 | 教学风格、沟通偏好 |
| 来源 | AI 观察学生表现 | 学生反馈 + AI 自我调整 |
| 影响 | 教什么、教多深 | 怎么教、用什么语气 |

Soul 的更新完全由 AI 自行完成，不给预设的调整规则。更新时把旧的 Soul 和新的对话历史一起丢给 AI，让模型自己分析、自己总结。维度名称和数值由 AI 自由定义。

更新 prompt 示例:
  "以下是学生与老师的对话记录，以及当前的教师人格画像(Soul)。
   请根据学生的反馈和互动表现，分析老师在教学风格上
   需要做哪些调整，输出更新后的 Soul 文本和评分。
   维度由你自己定义，保持格式一致即可。"

更新时机:
  ① Feedback Handler 触发时 — 学生主动反馈风格问题
  ② 每 5 次对话同步更新 — AI 回顾对话中学生的隐式情感信号

存储位置: students 表 soul 字段（JSON，与 profile 同级）
每次对话开始拼接到 System Prompt 中 → 所有 Handler 都感知到 Soul
```

### 4.6 教学忍耐与确认

```
每步教学输出 → 展示 → 确认学生是否理解 →
  理解了 → 下一步
  不理解 → AI 结合画像判断 →
    ├─ 已知薄弱点 → 降低忍耐，直接给提示或换角度讲解
    ├─ 偶发失误 → 正常引导
    └─ 反复不理解 → 直接给出本步结果，标记 [STUCK]知识点[/STUCK]
         → Harness 解析后存入记忆库

确认不是一次性的"你明白了吗？"，而是融入每步教学的确认。
学生回应"明白了"才算完成一步。不明确回应则视为未理解。

### 4.7 进度反馈

```
进度反馈是 Harness 在各阶段切换时向前端发送的状态提示，
让学生感知到 AI 的"思考过程"，提升交互体验。

不是独立的模块，而是贯穿 Harness 所有阶段的行为:
  Harness 每进入一个阶段 → 向前端推送一条进度消息 →
  前端显示在对话界面的状态栏或气泡上方

阶段 → 进度消息映射（示例）:

  Router AI 识别中     → "让我看看你问的是什么……"
  Generate（教学者）   → "原来是{学科}题，让老师想想……"
  Generate（切旗舰）   → "这个老师不太行，再为您换一个老师……"
  Review（前置审查）   → "老师检查一下这样讲合不合适……"
  Teach 第一步         → "来，我们一步一步看……"
  Teach 每步确认       → "这一步你理解了吗？"
  Teach 薄弱点卡住     → "这个地方你上次也不太熟，我们换种方式讲"
  Teach 偶发卡住       → "没关系，我们换个角度看看"
  Verify 开始          → "让老师看看你算得对不对……"
  Verify 正确          → "完全正确！👍"
  Verify 错误          → "这里有个小问题，你看……"
  Concept 开始         → "这个概念的来龙去脉是这样的……"
  Review 开始          → "老师看看你之前哪些地方不太熟……"
  Correct 开始         → "让老师看看是不是讲错了……"
  Correct 确认错误     → "你说得对，是老师讲错了，抱歉！"
  Correct 确认无误     → "老师检查过了，这个步骤是对的"
  Feedback 开始        → "老师知道了，下次注意……"
  Save 记忆库          → "老师记下来，下次复习的时候提醒你"

进度消息由 Harness 硬编码映射，不经过 LLM 生成。
这样保证消息即时显示（不等待 API 响应）。
```

### 4.8 对话持久化与话题切换

```
持久化策略:
  老师每回复学生一次（每步 Teach 审查通过后），
  立即将该条消息保存到 SQLite（messages 表）。
  → App 被杀/重启 → 从数据库恢复最近一次回复

话题切换策略:
  Analyze 阶段除识别学科和意向外，还需判断
  是否与当前对话属于同一话题:
  ├─ 同一话题 → 继续当前对话
  └─ 新话题 → 归档当前对话（conversations 表标记为 completed）
              → 创建新对话 → 清空内存中的 history
              → 从头开始 Analyze

话题切换判断依据:
  同话题: "还是刚才那道题，我想换个方法解"
         "那如果条件改成 XX 呢？"
  新话题: "先不说这个，我问你另一道题"
         "帮我看看这道英语题"（跨学科切换）
         "上次那个错题我还是不懂"（切换到历史话题）

这样每条 conversation 对应一个完整话题的教学过程，
history 长度自然受限于单个话题，不会无限膨胀。
```

## 五、记忆库（本地 SQLite）

| 表 | 用途 |
|----|------|
| students | 学生信息 + 画像(JSON) + Soul(JSON) + 对话计数器 |
| conversations | 会话记录 |
| messages | 消息记录 |
| mistake_book | 错题（题目、答案、错因分析、知识点） |
| knowledge_nodes | 知识点掌握程度（0.0~1.0） |
| knowledge_blacklist | 学生主动标记的超纲知识点黑名单 |

## 六、关键决策记录

| 决策 | 结论 |
|------|------|
| 审查者与教学者关系 | 完全独立对话 |
| 审查覆盖范围 | Generate 完整回答 + Teach 每步输出 |
| 忍耐值控制 | AI 自行判断，画像驱动 |
| 学生画像维护者 | AI 每 5 次对话自主更新 |
| 画像存储 | students 表 JSON 字段 |
| 教师 Soul | AI 自行维护的教学人格画像，旧 Soul + 新对话历史丢给 AI 自行分析更新，无预设维度规则 |
| Soul 更新时机 | Feedback Handler 触发 或 每 5 次对话同步更新 |
| 学科识别 | 只返回学科名，不细分模块 |
